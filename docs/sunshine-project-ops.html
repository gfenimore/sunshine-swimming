Section 2
Git Workflow

Branch Strategy

Single branch: main. No feature branches for now — the team is one person + one AI. Branching adds overhead without value at this scale.

Revisit when: Amanda's project goes live and we start a second client deployment. At that point, consider main + client/sunshine-swimming branches.

Commit Message Format

type: short description Types: feat: New feature or capability fix: Bug fix spec: Specification document added or updated docs: Documentation (README, architecture, etc.) refactor: Code restructure without behavior change test: Test data, test scripts, test harness ops: Deploy config, CI, project ops data: Data scripts or processed data Examples: feat: rebuild intake form from spec v1.0 fix: error clearing on select fields spec: add form specification document ops: add GitHub Pages redirect data: generate LOTR test import CSV

Commit Cadence

When to CommitExample
After a spec is approvedspec: intake form specification v1.0
After a feature is built + tested locallyfeat: rebuild intake form from spec
After a bug is fixed + verifiedfix: Start Over link styling
After docs are updateddocs: update architecture decision log
After a test pass is completedtest: Stage 1 routing scenarios pass
Never commit: half-finished work, untested changes, or "WIP" commits. Every commit on main should be a working state.

Push = Deploy

GitHub Pages is configured to serve from main branch, root folder. Every git push triggers a deploy. The live URL is:

https://gfenimore.github.io/sunshine-swimming/

This redirects to src/form/index.html via the root index.html.

Consequence: Pushing to main = deploying to the public URL. If Amanda has the link, she sees changes immediately. Don't push broken code.

Environment Switching

The form has two config variables at the top of the script. To switch environments, update both:

// DEV const ZAPIER_WEBHOOK_URL = 'YOUR_ZAPIER_WEBHOOK_URL_HERE'; const FORM_MODE = 'dev'; // STAGING const ZAPIER_WEBHOOK_URL = 'https://hooks.zapier.com/hooks/catch/1830145/uvk2o44/'; const FORM_MODE = 'staging'; // LIVE const ZAPIER_WEBHOOK_URL = 'https://hooks.zapier.com/hooks/catch/XXXXXXX/XXXXXXX/'; const FORM_MODE = 'live';
Current limitation: We have one file and one branch. Switching environments means editing the config and pushing. For now this is acceptable — the form is in active development, not serving real clients. When we go live, we'll need a build step or separate deployment for production.

Section 4
Change Management

The Rule

No code changes without an approved spec.

This is the rule that prevents us from going in circles. Every change follows this process:

Change Process

1. IDENTIFY → What needs to change and why? 2. SPEC → Claude updates the relevant spec document 3. REVIEW → Garry reviews the spec change in the artifact panel 4. APPROVE → Garry says "approved" or requests modifications 5. BUILD → Claude implements exactly what the spec says 6. TEST → Garry tests locally (DEV mode) 7. COMMIT → Garry runs: git add . && git commit -m "..." && git push 8. VERIFY → Garry checks GitHub Pages URL to confirm deploy

Change Types

TypeSpec Required?Process
New featureYes — add to form specFull process: spec → approve → build → test → commit
Bug fixNo — but describe the bug and fix before implementingDescribe → agree → fix → test → commit
Content change (copy, labels)No — but confirm the exact textState the change → agree → implement → verify
Config change (webhook URL, mode)NoState what's changing → implement → verify
New doc/artifactNoCreate → review → save to repo

Artifact → Repo Workflow

Claude creates artifacts in the chat. They need to get into the repo. The process:

  1. Claude creates an artifact (HTML doc, spec, plan) in the right panel
  2. Garry reviews in the panel
  3. If approved: Claude writes the file to Garry's filesystem via MCP
  4. Garry commits and pushes
Key insight: Claude has TWO file systems — the container (ephemeral, resets between sessions) and Garry's machine (via Filesystem MCP, persistent). Final deliverables must be written to Garry's machine, not just the container.

Section 6
Operating Rules

Rules for Claude

  1. Spec before code. No form changes without an approved spec. No exceptions.
  2. State what you'll change before changing it. List files, describe changes, get a "go" before editing.
  3. Don't guess about platform UIs. If you don't know what a Zapier or HoneyBook screen looks like, ask Garry to screenshot it.
  4. Write to Garry's filesystem, not just the container. The container resets. Garry's machine persists.
  5. Use artifacts for reviewable content. Specs, plans, and docs go in the right panel — not inline in chat.
  6. One thing at a time. Don't batch 5 changes in one edit. Each change is discrete, testable, and committable.
  7. Field names are sacred. Never rename a webhook payload field without explicit discussion. Zapier mappings break silently.
  8. Admit when you don't know. "I'm not sure how HoneyBook handles this" is better than a wrong answer that wastes 2 hours.

Rules for Garry

  1. Test locally before pushing. Open the file in the browser, verify the change works, then commit.
  2. Commit after every verified change. Don't accumulate uncommitted changes across sessions.
  3. Screenshot when stuck. A screenshot of the actual UI is worth 1000 words of description.
  4. Push back on Claude. If instructions don't make sense or reference things that don't exist, say so immediately.
  5. One environment at a time. Don't test in STAGING while also making DEV changes. Finish one, commit, then switch.

Rules for Both

  1. Never skip testing. Every change gets verified before the next change starts.
  2. Document surprises. When a platform does something unexpected, add it to docs/architecture.md decision log.
  3. Protect Amanda's data. Real client data never goes in git. Test data uses LOTR names. Garry's Gmail for test emails, never Amanda's until go-live.
  4. When in doubt, stop and align. 5 minutes of discussion saves 2 hours of rework.
