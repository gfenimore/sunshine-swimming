Sunshine Swimming · Form Spec v1.0

Intake Form Specification

Complete spec for the Sunshine Swimming custom intake form. Approve each section, then I rebuild from scratch to match exactly. No ad hoc changes.

Section 1
Overview

Purpose

A mobile-first, multi-step HTML form that qualifies swim lesson leads before they reach HoneyBook. Unqualified leads are declined inline with a warm message. Qualified leads are submitted to a Zapier webhook.

Technical Constraints

  • Single self-contained HTML file (no build tools, no framework)
  • No external dependencies except Google Fonts
  • Hostable anywhere: GitHub Pages, Netlify, WordPress embed
  • Zapier webhook URL configurable via a single variable at top of script
  • Test mode auto-enables when webhook URL is the placeholder value

Branding

  • Color palette: ocean blues (#0c4a6e → #bae6fd), white card, light sky background
  • Fonts: Fraunces (serif, headings), Nunito (sans, body/fields)
  • Tone: warm, friendly, personal — swim coach for toddlers, not enterprise SaaS
  • Logo: placeholder emoji (🏊‍♀️) until Amanda provides her logo file
  • Wave SVG divider between header gradient and form body

Webhook Payload Field Names

These are the JSON keys sent to Zapier. They must match exactly what Zapier sees.

full_name → Parent/Guardian name email → Email address phone → Phone number address → Home address swimmer_name → Swimmer's first name swimmer_age → Age range dropdown value previous_lessons → Yes / No / A little comfort_level → 1-5 scale dropdown value medical → Medical/allergies (optional) injuries → Prior injuries (optional) additional_swimmers→ Additional swimmers (optional) lesson_location → Alt lesson location (optional) location → Service area dropdown value lesson_type → Paid or nonprofit pool_heated → Yes / No / I don't know / Neighborhood preferred_times → Schedule preferences (optional) referral_source → How heard (comma-separated) anything_else → Open notes (optional) lead_type → "qualified" or "nonprofit" tag → same as lead_type submitted_at → ISO timestamp form_version → "1.0"

Section 4
Routing Logic

Decision Tree (evaluated on Step 1 "Continue" click)

IF location = "Outside Manatee County" → Show declineArea panel → Hide progress bar → NO webhook fires → STOP ELSE IF pool_heated = "No" → Show declinePool panel → Hide progress bar → NO webhook fires → STOP ELSE IF lesson_type = "Free water safety program (nonprofit)" → Skip Step 2 (swimmer details) → Go directly to Step 3 (contact info) → On Step 3 submit: send to webhook with lead_type = "nonprofit" → Show nonprofitThanks panel → STOP ELSE (qualified paid lead) → Go to Step 2 → Step 3 → Step 4 → On Step 4 submit: send to webhook with lead_type = "qualified" → Show thankYou panel → STOP
Key rule: Routing is evaluated in the order above. Location is checked first, then pool, then lesson type. A lead that is outside the area AND has an unheated pool gets the area decline, not the pool decline.

Section 7
Submission Logic

Qualified Lead Submission

  1. Gather all field values into a JSON object (using webhook key names from Section 1)
  2. Add lead_type: "qualified" and tag: "qualified"
  3. Add submitted_at (ISO timestamp) and form_version: "1.0"
  4. If TEST_MODE: log to console, fake 800ms delay, then show thank-you
  5. If live: POST to ZAPIER_WEBHOOK_URL with Content-Type: application/json and mode: 'no-cors'
  6. On completion: show thankYou panel, hide progress bar

Nonprofit Lead Submission

  1. Gather ONLY: location, lesson_type, pool_heated, full_name, phone, email, address, lesson_location
  2. Add lead_type: "nonprofit" and tag: "nonprofit"
  3. Add submitted_at and form_version
  4. Swimmer fields are NOT included (they were never shown)
  5. Submit to same webhook URL
  6. On completion: show nonprofitThanks panel, hide progress bar

Declined Leads

Declined leads are NEVER submitted. No webhook fires. No data leaves the browser. The decline screen is shown inline within the form card.

Amanda's Review Access

When we share the form with Amanda for testing, she needs to be able to review the policies and content embedded in the form (welcome text, decline messages, referral list, pool temperature requirements). The testing instructions we provide Amanda must include:

  • A checklist of every message/policy in the form for her to approve or edit
  • Clear instructions on how to request changes (reply to email, or shared doc)
  • Specific callout of the session price ($50 vs $60 — needs her confirmation)
  • Placeholder referral swim schools — she must provide real names
  • Amanda's phone number on the pool decline screen — confirm it's correct
Section 8
Configuration & Test Mode

Configurable Variables

// Top of the <script> block: const ZAPIER_WEBHOOK_URL = 'YOUR_ZAPIER_WEBHOOK_URL_HERE'; const FORM_MODE = 'dev'; // 'dev' | 'staging' | 'live'

Two variables to set. FORM_MODE controls the banner and logging behavior. ZAPIER_WEBHOOK_URL controls where data is sent. Both must be updated together when moving between environments.

Operating Modes

The form has three distinct operating modes. The mode is determined by the webhook URL configuration.

ModeWhenWho Uses ItBehavior
DEV Mode Webhook URL = placeholder string
'YOUR_ZAPIER_WEBHOOK_URL_HERE'		 Garry, during development Yellow banner: "🧪 DEV MODE". Submissions log to browser console only. "Run Test Suite" button available. No data leaves the browser. All routing/validation works normally.
STAGING Mode Webhook URL = Zapier test webhook URL Garry + Amanda, during testing Blue banner: "🔵 STAGING — Test submissions go to Zapier test environment". Data DOES submit to Zapier. Used for end-to-end testing before go-live. Amanda uses this to verify the form feels right and review content.
LIVE Mode Webhook URL = Zapier production webhook URL Amanda's real clients No banner. No test indicators. Clean production form. Data submits to the live Zapier zap connected to Amanda's HoneyBook.

Mode Detection Logic

const ZAPIER_WEBHOOK_URL = '...'; // Mode detection const DEV_MODE = (ZAPIER_WEBHOOK_URL === 'YOUR_ZAPIER_WEBHOOK_URL_HERE'); const STAGING_MODE = ZAPIER_WEBHOOK_URL.includes('/hooks/catch/') && ZAPIER_WEBHOOK_URL.includes('test'); // LIVE_MODE = !DEV_MODE && !STAGING_MODE
Open question: Zapier doesn't distinguish test vs production webhook URLs natively. We may need a separate config flag like const FORM_MODE = 'dev' | 'staging' | 'live' instead of auto-detecting from the URL. To be decided during implementation.
